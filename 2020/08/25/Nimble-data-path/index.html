Nimble 协议栈数据通路

发表于 2020-08-25 更新于 2021-09-25 分类于 通信

NimBLE 是 Apache 基金会下一个开源的蓝牙5.0协议栈，具备完整的 Host 及 Controller 层。资源占用少，支持蓝牙 5.0 特性，也支持 Mesh 等功能。

下面简单分析了 NimBsLE 的数据通路。

目录结构

内存的分配和释放

Nimble 的内存管理采用固定分区分配的方法，即将内存划分成若干个固定大小的块，用户申请的时候分配其满足其大小需求的一个或多个块。

另一种常用的分配方案为动态分区分配。即需要一个空闲表或者空闲链来记录目前系统中空间的内存区域。在内存分配时，需要查找空间表或空闲链找到一块内存分配给当前进程。

相对于动态分区分配，固定分区分配有一些优缺点：

  • 优点
    • 内存管理简单，每次分配只需要给用户足够数量的内存块。
    • 不易碎片化，每个内存块独立，分配给用户的内存块不必连续。
  • 缺点
    • 可能造成内存浪费，即使用户申请一个字节的内存，实际分配的也为一整个块。
    • 数据处理繁琐，多个块物理s上不连续，memcpy 等操作不方便。

下面我们看一下 Nimble 里的具体实现。porting/nimble/include/mem/mem.h 里定义了内存池初始化相关的函数：

  • mem_malloc_mempool 分配 num_blocks * (block_size 向上取整到 4/8/16 的倍数) 大小的内存。并将其切割成 (block_size 向上取整到 4/8/16 的倍数)大小的块，挂在 mempool 的链表上。
  • mem_malloc_mempool_ext 同 mem_malloc_mempool，只是多注册里一个回调函数，当 mempool 里的块被释放时会调用该回调函数。
  • mem_malloc_mbuf_pool 初始化 mempool 后，将其放到 mbuf_pool 里，并将 mbuf_pool->omp_databuf_len 赋值为 buf_len - sizeof(struct os_mbuf)。
  • mem_malloc_mbufpkt_pool 同 mem_malloc_mbuf_pool，只是初始化 mempool 的时候把块大小加上了 struct os_mbuf_pkthdr 的大小。
  • mem_init_mbuf_pool 同 mem_malloc_mbuf_pool，只是由用户给定物理内存，不自动分配。

porting/nimble/include/os/os_mbuf.h 里定义里一些内存分配和释放的函数：

  • os_mbuf_get 从一个 mbuf_pool 里获取一个块，要求用户将要使用的头信息（leadingspace）大于内存块里的有效大小。
  • os_mbuf_append 将一段数据拷贝到 mbuf 里，当前内存块大小不够时会自动分配新的内存块并用链表挂在一起。
  • os_mbuf_free 将一个内存块释放并放回内存池
  • os_mbuf_free_chain 将一串内存块全部释放并放回内存池

用户和协议栈的数据交互

GAP

注意：

  • 该回调函数每个连接单独配置。
  • ADV，SCAN 等会单独配置，当连接建立的时候，该连接会自动沿用 ADV/SCAN 的回调函数。之后用户可调用接口 int ble_gap_set_event_cb(uint16_t conn_handle, ble_gap_event_fn *cb, void *cb_arg); 覆盖。
  • 若多处注册的回调函数为同一实体，参数 arg 可用作区分回调来自于谁。

L2CAP

其中 event type 有：

  • BLE_L2CAP_EVENT_COC_CONNECTED
  • BLE_L2CAP_EVENT_COC_DISCONNECTED
  • BLE_L2CAP_EVENT_COC_ACCEPT
  • BLE_L2CAP_EVENT_COC_DATA_RECEIVED
  • BLE_L2CAP_EVENT_COC_TX_UNSTALLED
  • BLE_L2CAP_EVENT_COC_RECONFIG_COMPLETED
  • BLE_L2CAP_EVENT_COC_PEER_RECONFIGURED

event 的内容里有 ble_l2cap_chan 项, 用户由此通过函数 ble_l2cap_get_chan_info 获得具体信息。

ble_l2cap_chan 对外仅是一个代号，没有具体含义。但对内是一个包含各种信息的结构体。

GATT

GATTS 可通过 ble_gattc_notify/ble_gattc_indicate 向 client 端发送数据。

  • 函数名称有误，应该为 ble_gatts_notify/ble_gatts_indicate。Pull request

Servers 端每个 characteristic 和 descriptor 的结构（ble_gatt_chr_def/ble_gatt_dsc_def）里会有一个 ble_gatt_access_fn *access_cb 属性，用来通知用户有 client 读写。

回调函数返回后，协议栈会自动释放内存。如果用户想保留该 buffer 的话，可以将参数里的指针置为空。

GATTC 通过 ble_gattc_read/ble_gattc_write 读写 server 端的值（非阻塞）。成功后通过 callback 通知用户。

GATTC 如何接收 server 端发来的 notify/indicate？？？

回调函数返回后，协议栈会自动释放内存。如果用户想保留该 buffer 的话，可以将参数里的指针置为空。

HOST 和 CONTROLLER 的数据交互

nimble/include/nimble/ble_hci_trans.h 里

ble_hci_trans_ll_evt_tx 和 ble_hci_trans_ll_acl_tx 负责 controller 发送 event 和 ACL 数据；
ble_hci_trans_cfg_ll 负责 controller 注册收发 CMD 和 ACL 的 callback。

ble_hci_trans_hs_cmd_tx 和 ble_hci_trans_hs_acl_tx 负责 host 发送 command 和 ACL 数据；
ble_hci_trans_cfg_hs 负责 host 注册收发 event 和 ACL 的 callback。

上诉函数的实现在 nimble/transport/xxx/src/xxx.c 里。可通过 UART，USB，socket，ram 等传输 HCI， 根据传输介质的不同实现不同。大部分的实现为将数据直接送给传输介质。

若 controller 和 host 在同一设备上，则可采取 ram 的方法传输，即 TX 函数直接调用对方注册相对应的 RX callback，将数据的指针直接传送给对方，内存的释放由接收方负责。该方法实现简单，没有数据的传输和拷贝带来的开销。例如：

nimble host 发送 CMD 时，采用的是同步的方法，同一时间只有一个 HCI CMD 的交互，发送 HCI CMD 后会立刻等待 controller 的回复。

  • 该接口为阻塞接口，效率较低。若需求高效率可考虑改为异步接口。
